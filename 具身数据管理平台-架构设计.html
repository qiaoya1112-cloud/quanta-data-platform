具身数据管理平台 — 架构设计文档

版本 v1.0 · 2026-06-04
范围:带预览 UI 的 Web 数据管理平台,自建(LAS 仅作竞品参考)

0. 背景与目标

现状

  • 数据格式:LeRobot v2.1 标准。每条 recording = dataset.json(业务元信息) + meta/(info/tasks/episodes/episodes_stats) + data/*.parquet(双臂 14 维 state/action 时序) + videos/*.mp4(3 路相机:cam_high / cam_left_wrist / cam_right_wrist)。机器人型号 moz1,fps=30。
  • 原始数据:已在 TOS,天然 recording 粒度,路径 recordings/{recording_id}/...,通过预签名 URL 下载。
  • 处理流程:服务端 Quanta API → 导出(普通线 / DAgger 线)→ 切 train/val → 配采样权重 → 统计 + 质检 → 喂训练。全程是 CLI 脚本 + 手工拼装的 {数据集:权重} JSON。

痛点

没有中央目录、血缘靠手工 JSON、质检结果不落库、无版本管理、纯命令行无预览。

目标

把"脚本 + 手工 JSON"升级为"中央目录 + 血缘可追溯 + 可视化预览 + 一键编排 + 质量门禁"的 Web 平台。脚本不重写,包装成平台的算子。

1. 核心心智模型:两层视图

借鉴 LAS 的分层,平台底层是同一套 TOS 存储 + 同一套元数据服务,对外提供两层视图:

层视角抽象单位实现关心的事
湖表层(数据湖)数据工程 / 可筛表 / 字段 / 分区recording manifest 表,按 collection_id / robot_id / task 分区有哪些数据、可 SQL 筛、权限、可审计
数据集层AI / 训练样本 / 模态 / 标签LeRobot 数据集,切 train/val/eval样本数、模态、配方、喂模型

术语纪律:"Catalog"专指湖表元数据层;管理数据集的那层称「数据集管理 / 数据集详情」,不叫 Catalog,避免歧义。

关键设计:湖表 = manifest 表(存指针),不是把信号灌进表。每行一条 recording,列 = 元数据 + TOS 路径 + episode_stats 摘要;mp4/parquet 大文件留在 TOS。

4. 功能模块

模块 1 · 原始数据湖(湖表层)

  • recording 索引表:从采集侧 / dataset.json 抽元数据建 manifest 表,大文件留 TOS。
  • 多维筛选(faceted):task × 设备 × collection × 时间 × 数据类型(采集/test/dagger),task 是默认聚合维度但非唯一。
  • recording 级预览(见模块 3):预签名 URL 直接流式播放,无需下载到平台。
  • 生命周期:坏帧/废采集标记 quarantined,从可选池排除,不物理删除。

模块 2 · 数据集生成(筛选 → 配方 → 切分 → 质检)

借鉴 LeRobot FILTERING 的三层清洗结构:

  1. 机械式筛选:时长 / 帧率等硬指标,用滑块/范围。
  2. 统计式筛选:动作量最小 / 最抖的排行榜 + flag(可批量 Flag all)。
  3. 诊断式筛选:整体健康自动打标签(如 smoothness 等级)。

核心产品原则:诊断/执行分离——平台只做诊断 + 标记(flag),绝不一键删除;标记结果导出成操作清单,由人确认执行。理由:数据是具身项目最值钱的资产,误删成本巨大。

  • 配方(Recipe):GROUP_RULES / GROUP_CONFIG / 采样权重从硬编码变成结构化可编辑,改完实时预览"权重占比 vs 帧数占比 vs 实际采样占比"三栏对比。
  • 生成 = 不可变快照:输入 recording_ids + 配方 + seed → 冻结的 dataset 版本,可复现。
  • eval 泄漏防护:选中 recording 若已属某评测集,红色告警。

「验证」与「质检」的区分(脚本对应)

现有算法脚本里的"验证、质检"内容是两类不同的功能,落到不同子环节,不要混为一谈:

  • 验证(Validation)= 配方配得对不对 → 属于配方配置子环节
  • adjust_sampling_ratio.py:算采样权重(算法本体)
  • analyze_sampling_frames.py:验证"实际采样占比是否符合设定",给配方所见即所得的反馈
  • 质检(QA)= 数据本身对不对 → 属于质检门禁子环节,背后调 DOCTOR 通用能力
  • check_chinese.py:标注合规(中文混入)
  • check_evalset_split.sh:数据完整性(meta 帧数 vs parquet 行数 / 缺帧)
  • 统计(描述性) → 跨到数据集管理的元数据/构成分析
  • dataset_statistics.py:帧数/时长/prompt/recording 统计

数据集生成
├─ 三层筛选 ├─ flag 标记
├─ 配方配置(采样权重/分组)
│    ├─ adjust_sampling_ratio.py   ← 算权重
│    └─ analyze_sampling_frames.py ← 【验证】占比符不符合预期
├─ train/val/eval 切分
├─ 质检门禁校验  ← 调 DOCTOR 能力
│    ├─ check_chinese.py           ← 【质检】标注合规
│    └─ check_evalset_split.sh     ← 【质检】数据完整性
└─ 快照生成
一句话:质检 → DOCTOR 门禁(拦坏数据);验证 → 配方配置的采样自检(确认配方生效)。dataset_statistics.py 属描述性统计,沉淀到数据集管理。

模块 3 · recording 级预览(探索性,人脑判断)

预览三件套是基线(否则工程师不用):

  • state/action 叠加曲线:leftarm_state_*(实线)vs leftarm_cmd_*(虚线)叠在同图,一眼看控制延迟/跟随质量。
  • 视频↔曲线同步光标:一条竖线串起 3 路相机 + 14 维曲线。
  • 键盘流:Space 播放、↑↓ 切 episode。
  • 曲线按"相邻关节"分组(左臂 7 关节一组、右臂一组、双 gripper),不按左右臂。

进阶:

  • FRAMES 首末帧网格:低成本高 ROI,判断"任务到底成没成"(首帧初始态一致性 / 末帧完成态)。建议 P1 即做。
  • 3D Replay + Trail:URDF 白名单 + Joint Mapping 桥接任意列名;末端轨迹 Trail 看运动模式;action vs state 双源重建。(可后置)

模块 4 · DOCTOR 质检门禁(结构化质量保障)

把质检产品化成红绿灯门禁:

  • PASS / WARN / FAIL 三档 + 顶部一句话健康总结 + 分项详情 + JSON 输出(可接 CI)。
  • 流程门禁:PASS → 可进训练队列;WARN → 需人工确认;FAIL → 打回修。
  • 检查项(很多可用现成 episode_stats 零成本算):
  • frozen action(某维 std≈0,如本数据 psi 全 0)
  • dropped frames(meta 帧数 vs parquet 行数,复用 check_evalset_split.sh)
  • clipped values(贴边界)
  • 相机分辨率一致性(都应 240×320)
  • 中文混入标注(复用 check_chinese.py)
  • 采样占比偏差(复用 analyze_sampling_frames.py)

模块 5 · 数据集管理 / 数据集详情

  • 元数据:基础(robot_type/version/tasks)→ 规模(总帧数/episodes/fps/总时长)→ 模态(相机分辨率)→ 时序(episode 长度分布直方图)。方差/分布比均值有信息量。
  • 构成分析(配方):按 group/task/设备 的帧数占比环图 + 权重 vs 实际采样对比。
  • 质量数据:DOCTOR 红绿灯 + 明细。
  • 血缘:↑由哪些采集/test/dagger 生成;↓被哪些训练任务引用。

模块 6 · 发布与引用(TOS)

  • 发布 = 冻结为不可变版本:生成带版本号的 manifest,tos://.../datasets/{name}/{version}/。
  • 去重上传:原始 recording 已在 TOS,优先用 TOS 内 copy / 软链清单,不重传大文件。
  • 引用登记:训练任务启动回调平台登记 dataset_version ← training_job,血缘"被谁引用"才有数据。

模块 7 · 任务中心(编排)

  • 导出/切分/采样/质检任务统一队列:状态、日志、耗时、产出物。
  • 流水线模板:导出→切分→采样→质检 串成 DAG,一键触发、有状态、可重跑单阶段。
  • 现有脚本包装成算子,不重写。

5. 训练集 / 评测集策略

结论:要分,且分成两类各司其职,不要只做一个随机 95/5。

具身策略真实评测靠真机/仿真 rollout 成功率,离线 val loss 与成功率相关性弱,故单纯随机切的 val 集对选模型价值有限。

训练内验证集 (val)独立评测集 (eval / benchmark)
目的监控过拟合、看 action MSE、早停衡量泛化、定真机评测场景
切分粒度episode/recording 级随机切(95/5)task/scene/collection 级整体留出
关键约束同一 recording 的帧绝不跨 train/valrecording 永不进任何训练集
来源从训练数据切最好专门采集的 collection

两个强约束:

  1. 切分必须在 recording/episode 级,不能 frame 级(相邻帧高度相关,frame 级切高估性能)。
  2. 评测集做成数据湖一等公民,全局留出;生成任何训练集时平台自动校验未碰评测集 recording。

数据集类型枚举:train / val / eval-benchmark。

6. 训推评闭环(自建独有优势)

LAS 等开源/通用工具做不到,应显式设计进血缘:

  • 从评测反推数据:某 policy 在哪些 episode 失败率高 → 反查这些 recording 特征 → 定位数据问题(复用"数据集→训练任务"血缘的反向边)。
  • 本体专属阈值:沉淀"moz1 在某任务下动作 σ 的正常范围",比通用阈值有价值。
  • 跨数据集对比:配方/质量/抖动横比。
注:训练超参建议(action chunk 长度等)不在本期范围,平台职责聚焦"管数据 + 控质量",不碰训练配置。
