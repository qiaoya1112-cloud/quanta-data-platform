# Quanta · 具身数据管理平台 PRD

| 项 | 内容 |
|---|---|
| 产品名称 | Quanta（具身数据管理平台） |
| 文档版本 | v1.0（基于交互 Demo 整理） |
| 定位 | 面向具身智能 / 机器人训练数据的「原料 → 成品 → 交付」全流程数据管理平台 |
| 数据格式 | LeRobot v2.1（本体 moz1，fps 30，三路相机，双臂 state/cmd） |
| 适用角色 | 数据工程师、算法工程师、质检/标注人员 |

---

## 1. 背景与目标

### 1.1 背景
具身智能模型训练高度依赖大规模、多任务、多场景的真机采集数据。原始采集数据（recording）数量大、质量参差，从「采集落盘」到「喂入训练」之间存在大量人工：盘点筛选、质检体检、配方配比、切分采样、合规检查、版本管理。缺少统一平台时，这些环节散落在脚本与人工沟通中，难以复现、难以追溯。

### 1.2 产品目标
- **统一原料管理**：以数据湖（manifest 清单）统一盘点、检索、预览全部 recording。
- **可复现的成品**：以数据集（不可变版本快照）作为喂入训练的标准交付物，全链路可追溯。
- **自动化加工**：把数据处理脚本沉淀为「算子」，编排为可复用、可调度的「任务（流程）」。
- **降低使用门槛**：提供 AI 助手（自然语言→SQL）与配方助手（按比例随机采样）辅助筛选与组数据集。

### 1.3 核心设计原则
- **两层架构**：数据湖（可变、recording 级、元数据清单）↔ 数据集（不可变、版本化快照）。
- **三层配置**：算子（模板）→ 节点（实例）→ 流程（上下文），契约与默认在算子、取值与绑定在节点、串联与全局在流程。
- **recording ↔ episode 1:1**：一条采集 recording 对应一个 episode。

## 2. 名词与数据模型

### 2.1 术语
| 术语 | 含义 |
|---|---|
| recording | 一次真机采集的原始数据单元；与 episode 1:1。 |
| episode | LeRobot 数据单元；本平台等价于一条 recording。 |
| 数据湖 | 全部 recording 的清单层（manifest），可变、记录级，承载元数据与生命周期。 |
| 数据集 | 由筛选 / 加工产出的**不可变版本快照**，喂入训练 / 评测的标准交付物。 |
| 算子（Operator） | 一个「原子处理能力」的封装（= frontdesk_scripts 中一个脚本），可被任务编排。 |
| 任务 / 流程（Pipeline） | 由多个算子编排成的可复用、可调度的 DAG。 |
| 节点（Node） | 算子在某个流程中的一次实例化。 |
| 执行实例（Run） | 一次任务执行记录。 |

### 2.2 recording / episode 元数据

每条 recording（= episode）的元数据字段：

| 字段 | 类型 | 说明 | 示例 |
|---|---|---|---|
| `recording_id` | string | 全局唯一 id | `#120489` |
| `task` | enum(task_id) | 所属采集任务 / prompt | `t1`（Clean the whiteboard） |
| `collection_id` | string | 采集批次 / 采集任务 task | `#3635` |
| `capture_device` | enum | 采集设备 | `Moz` / `uDAS` |
| `operator` | string | 采集操作员 | `Lance Li` / `Wei Zhang` / `Min Chen` |
| `type` | enum | 采集类型 | `normal` / `dagger` / `test` / `eval` |
| `qa_status` | enum | 质检结论 | 合格(PASS) / 不合格(FAIL) / 操作失误(WARN) |
| `frames` | int | 帧数 | `1434` |
| `duration_s` | float | 时长（秒，= frames/fps） | `47.8` |
| `collected_at` | datetime | 采集时间 | `2026-05-21 10:12` |
| `tags` | array | 标签（场景 + 技能 + 质量，见 §2.5） | `[白板区, 擦拭, 高质量]` |

**采集结构 / 本体（LeRobot v2.1）**
- 本体：`moz1`，`fps = 30`。
- 三路相机：`cam_high`、`cam_left_wrist`、`cam_right_wrist`（240×320）。
- 特征：`leftarm/rightarm_state_joint_pos`、`leftarm/rightarm_cmd_joint_pos`，轨迹维度 X/Y/Z/r/p/y/G。
- 文件：`episodes.jsonl`、`tasks.jsonl`、parquet（帧数据）。

### 2.3 数据集元数据

| 字段 | 类型 | 说明 |
|---|---|---|
| `dataset_id` / `name` | string | 数据集标识与名称 |
| `version` | string | 当前版本号（如 `v3`） |
| `status` | enum | `生效中` / `待发布` / `已归档` |
| `quality` | enum | DOCTOR 体检结论 PASS / WARN / FAIL |
| `episodes` / `frames` | int | episode 数 / 总帧数 |
| `recordings` | array | 组成该数据集的 recording id 列表 |
| `tasks` | array | 涉及 task / prompt |
| `folder` | enum | 业务语义目录（预训练 / 后训练 / 自定义） |
| `used_by` | array | 下游引用（训练 / 评测任务） |
| `created_by` / `created_at` | string | 创建人 / 创建时间 |

**版本记录（DS_VERSIONS）**：`version`、`说明`、`recording 数`、`创建时间`、`创建人`、`status`（生效中 / 待发布 / 已归档）。版本一经被训练引用即不可变。

### 2.4 业务语义目录
数据集按**业务语义**而非 train/eval 分组：`预训练`、`后训练`、`自定义01`、`自定义02`（用户可新建 / 重命名 / 删除目录，支持上级目录）。

### 2.5 标签体系（标签管理）
- **筛选用标签**（recording 级，三维度）：
  - 场景：白板区 / 桌面 / 绿植区
  - 技能：擦拭 / 抓取 / 放置 / 倒水
  - 质量：高质量 / 待复核（由质检结论派生）
- **标签管理体系**（资产 / 标签管理）：动作标签（拿取、放置、移动、旋转…）、场景标签、本体标签等多维度叶子标签库，含分组规则与版本。

### 2.6 质检（DOCTOR 体检）规则
统一规则集（数据集质量报告与 episode 预览质检一致）：
1. Metadata & Format Compliance（info.json / 字段 schema）
2. Temporal Consistency（时间戳单调，无跳变）
3. Frame Integrity 掉帧（meta 帧数 = parquet 行数，check_evalset_split）
4. Frozen Action 冻结动作（无某维 std≈0）
5. Camera Resolution（三路相机 240×320 一致）
6. Annotation 合规（tasks.jsonl 无中文混入，check_chinese）

每条规则在数据集维度展示「通过 episode 比例」（PASS=100%，WARN/FAIL 按受影响 episode 计算）。

## 3. 算子 / 节点 / 流程 三层配置说明

> 核心原则：**契约和默认在「算子」，取值和绑定在「节点」，串联和全局在「流程」**，并允许「节点级覆盖算子默认」。

| 配置项 | 算子级（模板，只读引用） | 节点级（实例） | 流程级（上下文） |
|---|---|---|---|
| 运行镜像 | ✅ 默认镜像 | 可覆盖（钉版本 / GPU） | （可选）统一默认 |
| 入口脚本 / 命令 | ✅ 脚本本体 + 命令模板 | 填参后的实际命令 | — |
| 出入参 schema | ✅ 接口契约（请求参数 / 返回结果） | — | — |
| 入参取值 | 默认值 | ✅ 本节点实际取值 | 可引用流程全局变量 `${var}` |
| 入参来源 | — | 绑定上游产物 | ✅ DAG 连线决定数据传递 |
| 连线 / 拓扑 / 调度 / 版本 / 重试超时告警 | — | — | ✅ 纯流程级 |
| 资源 / 队列 | — | 可覆盖 | 流程默认 |

### 3.1 算子（Operator）
- 字段：名称、标识（英文唯一）、描述、是否启用、运行镜像、入口脚本/命令、请求参数（schema）、返回结果、分类、创建人。
- 算子 = `frontdesk_scripts` 中一个原子处理能力（如切分 `batch_split`、采样配比 `adjust_sampling`、采样验证 `analyze_sampling`、统计 `dataset_stats`、中文质检 `check_chinese`、完整性质检 `check_evalset`）。
- 新建 / 编辑 / 详情均为抽屉；编辑可「保存」或「保存为新版本」；新建时可勾选「同步创建任务」。

### 3.2 节点（Node）
- 自由画布上的一个算子实例。
- 节点配置面板分段：基本信息（名称 / 标识 / 描述，来自算子定义）、运行环境（镜像 / 入口脚本，来自算子定义）、出入参（入参取值可填、返回结果只读）。
- 入参取值支持来源标记：`算子默认` / `本节点覆盖`（编辑即切换）/ `引用流程变量`（自动填 `${target_datasets}`）。

### 3.3 流程（Pipeline / Task）
- 自由画布：左右走向，节点可任意拖动，锚点拖拽连线，支持并行 / 分支（非线性 DAG）。
- 流程参数：流程级全局变量（如 `target_datasets` 支持多个逗号分隔、`tos_bucket` 系统内置只读），被多个节点以 `${var}` 引用。
- 周期调度：支持多组调度组（+添加一组），每组含调度频率 / 执行时间 / Cron 表达式；含生效范围、默认运行参数、异常处理（失败重试 / 超时 / 失败通知）。
- 执行：手动触发弹窗填写本次运行参数后提交。

## 5. 各页面交互与规则详情

### 5.1 数据查询（/query）
**用途**：在数据湖中按复杂条件检索 recording，按 episode 平铺，支持一键组数据集。

**筛选区**（默认全部展开，无折叠）
- 第一行（加宽撑满）：任务 Task ID（文本，多个逗号隔开）、标签（多选，按场景/技能/质量分组）。
- 第二行（撑满）：质检结果（多选，默认「全部」）、采集设备（单选 Moz/uDAS）、操作员（多选）、采集类型（多选 normal/dagger/test/eval）、采集时间（区间）。
- 工具行（右对齐）：`AI 助手`、`配方助手`、竖线分隔、`重置`、`查询`。

**AI 助手（侧栏聊天，紫色 icon）**
- 同排推开、无遮挡：打开时左侧内容响应式收窄，面板为右侧独立列。
- 多轮对话：用户用自然语言描述 → 生成 SQL（气泡）→「执行查询」。发送按钮为输入框内 icon。

**配方助手（侧栏）**
- 展示当前筛选结果的**提示词构成**（按 prompt）与**标签构成**（按技能）比例条。
- **相对权重模型**：每类默认 ×1（保持原始构成），独立拖动调整某类权重（其余不变），细拖动条、主题色。
- 实时显示「采样后约 N 个 episode」（按各类「可用量 ÷ 占比」的最紧约束计算），「确认采样」按比例随机过滤。底部按钮右对齐。

**结果列表**
- 顶部提示：「默认展示 20 条，更多请精确查询 · 共 N 条命中」。
- 列：Episode、Recording、Task、标签、设备、类型、帧数、时长、操作（预览）。
- 分页：每页 20 条，底部页码切换；表格横向滚动不压缩列宽。

**预览（抽屉，宽与数据集详情一致）**
- 仅展示：三路相机视频占位 + 5 个 tab 轨迹（LeftArm / Torso / RightArm / Base / 3D Replay）。
- 不展示目录、基本信息、质检、可视化分析。

**用结果建数据集（居中弹窗）**
- 填写数据集名称、标识、选择目录 → 确认创建。

### 5.2 数据集管理（/datasets）
**布局**：左侧业务语义目录树 + 右侧数据集详情（默认选中第一个）。

**目录树**
- 业务语义分组（预训练 / 后训练 / 自定义…），目录可展开收起、管理（重命名 / 删除 / 移入）。
- `+ 目录`：弹窗（居中）填写目录名称、标识、上级目录。
- `+ 数据集`：宽弹窗（居中），顶部提示「可前往数据查询用 AI 助手和配方工具筛选数据一键生成」，含数据集信息 + 全套查询筛选条件（任务 id 一行、标签一行、采集类型+时间一行、质检结果+设备+操作员一行）。

**详情 Tab**：数据预览 / 构成分析 / 质量报告 / 血缘 / 版本 / 基本信息。
- 数据预览：左侧 episode 列表 + 右侧三相机视频 + 5 tab 轨迹 + 可视化分析（轨迹拖尾 / 速度·加速度 / 延迟·抖动）。
- 构成分析：Prompt 构成条形图 + 标签构成（场景/技能/质量三饼图 + 图例）。
- 质量报告：DOCTOR 体检 banner + 6 条规则，右侧展示每条规则的「通过 episode 比例」。
- 血缘：上游来源（采集任务 task）表 + 下游引用（训练/评测任务 + 引用版本）表。
- 版本：版本 / 说明 / recording 数 / 创建时间 / 创建人 / 状态（生效中 / 待发布 / 已归档），可发布 / 回退。
- 基本信息：涉及 task / 标签 / 采集批次 / 本体·帧率 / 相机。

**处理数据（居中弹窗，定高，可内部滚动）**
- 任务（单选已配置好的任务）→ 处理说明（在参数之上）→ 请求参数。
- 请求参数：结构化 / 代码 切换（按钮小巧）；切换不改变弹窗高度。
  - 结构化：参数名由所选任务的算子配置决定，**只读不可改、不可新增**，仅填写取值。
  - 代码：占满整行的代码输入框。
- 确认执行 → 生成新版本（版本记录 +1，执行记录 +1）。

### 5.3 算子管理（/operators）
- 列表（两行高）：名称、标识、描述、创建人、操作。
- 顶部筛选：搜索 + 创建人（多选）。
- 操作：`关联任务`（跳转任务管理并按当前算子筛选）、`编辑`。
- 新建 / 编辑 / 详情抽屉：基本信息（名称 / 标识 / 描述 / 是否启用）、执行配置（运行镜像 / 入口脚本）、接口定义（请求参数 / 返回结果，代码编辑器）。
- 新建抽屉底部：☑ 同步创建任务（默认勾选）；编辑底部：取消 / 保存 / 保存为新版本。

### 5.4 任务管理（/pipelines）
**列表**：名称、描述、算子数、周期任务数（徽标，hover 显示具体时间）、创建人、状态（启用/停用开关）、更新时间、操作（编辑 / 执行）。
- 顶部筛选：搜索 + 算子（多选）。
- 执行：弹窗填写运行参数后提交。

**编排（自由画布）**
- 顶栏：任务名 + 流程参数 / 周期调度 / 保存 / 执行（无「编排 / 执行记录」tab）。
- 画布：节点绝对定位可拖动，锚点拖拽连线，支持并行；点击节点出右侧配置面板。
- 节点配置面板：基本信息 / 运行环境（来自算子定义）+ 出入参（入参取值可填、可引用流程变量）。
- 流程参数弹窗：全局变量增删改（`target_datasets` 多值、`tos_bucket` 只读）。
- 周期调度弹窗：多组调度组（+添加一组，每组频率/时间/Cron）+ 生效范围 + 默认运行参数 + 异常处理。

### 5.5 执行记录（/runs）
- 每行 = 一次任务执行实例。
- 列：实例 ID、任务名称、产出数据集、状态（运行中/成功/失败）、触发方式（手动触发 / 定时触发）、触发人（手动=触发人，定时=任务创建人）、开始时间、耗时、操作（日志 / 重跑）。
- 顶部筛选：任务 / 状态 / 触发方式 / 搜索。

### 5.6 标签管理（/tags，资产）
- 多维度标签体系展示（动作 / 场景 / 本体等），含叶子标签、分组、版本。

## 7. 用户故事

### 数据工程师
- 作为数据工程师，我想**按任务、设备、质检结果、采集时间等条件检索 recording**，以便快速盘点符合要求的原料。
- 作为数据工程师，我想**用自然语言描述需求让 AI 生成 SQL**，以便不写 SQL 也能完成复杂筛选。
- 作为数据工程师，我想**预览某条 episode 的三相机视频与轨迹曲线**，以便确认数据质量。
- 作为数据工程师，我想**把筛选结果一键打包成数据集并归入业务目录**，以便交付给算法侧。

### 算法工程师
- 作为算法工程师，我想**按提示词 / 标签的相对比例调配数据集配方并随机采样**，以便控制训练数据的构成分布。
- 作为算法工程师，我想**在数据集上执行一个已配置好的任务（切分 / 配比 / 质检）并产出新版本**，以便迭代实验且可复现。

## 8. 非功能性与后续

- **可复现性**：数据集版本不可变；处理 / 配方均产出新版本并留痕（版本记录 + 执行记录）。
- **可追溯性**：血缘记录上游采集任务与下游训练/评测引用版本。
- **存储**：TOS（recordings/ 与 datasets/）。
- **后续方向**：训练 / 部署 / 评测模块（顶导已占位）、评测结果反推数据、权限与审计、Quanta API 数据接入（普通线 / DAgger 线）。
